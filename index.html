About

Glock Bird is a web-game on Itch.io that takes familiar mechanics of a bird flying between obstacles while adding its own twist by giving that bird a gun. The game features cutom-made sprites and music with inspiration from retro arcade games.

The Process

Glock Bird is my first project in the Unity Engine and it really challenged me to learn everything I could about Unity’s interface while also trying to make a fun experience. In my pursuit I learned project management, programming, and game design skills that will carry me a long way. Also, in adjacent to this game’s development I was also going through Unity’s “Create with Code” course on their website to make sure I was not missing any important concepts with Unity’s interface. I used resources such as Miro Board, Notion, and Google Docs to design the game. I also designed the music and sprites for the game. To do this I used FamiStudio, Pixilart, and Krita.

Cursed Dungeon

Previous Next

About

Cursed Dungeon is a retro-style arcade game where the player is trapped in an endless dungeon where every room looks the same! The player has to complete a timing challenge at every door in order to progress to the next room, and the challenges become more and more frantic as the player progresses. Everything from the art, the coding, and the design was done by me in creating this game. Also, The game was designed for the Boss Rush Game Jam 2025 on Itch, the theme being "spin."

Cursed Dungeon was made using Unity Engine and the art was made using Resprite for IOS. GitHub was used for version control and the code is open-source for anyone to view.

Designing the Game

I joined the game jam around 2 weeks before it was due, so I had to quickly come up with an idea and begin implementation as soon as possible. I drew up some project sketches and diagrams, which can be found posted on my Itch page. I knew I wanted to do something retro and dungeon-like because I am a huge fan of dungeon-crawler games. I settled on a quick idea I could implement with an eyeball spinning inside of a stone mask. I didn't wanna bog myself down with controls so I decided to only add one control, and that is to simply stop the eyeball when it lands on the lock sigil.

Challenges and Takeaways

The first week of development was spent mostly on implementation. I used component-oriented design whenever possible to try to keep my code modular and scalable. A challenge I faced with this phase was sacrificing some intuitive design-choices for time. Time became my biggest hinderance when it came to this project because of classes. Still, my code turned out to be passably scalable by the time the game released.

The second week of the game's development was entirely spent on looks and polishing. I had to rethink how I approached the scripting and animations several times to work with new things I wanted to add. I learned a ton about making scripting that works with game animations and sounds through this process.

Overall this project taught me a lot about game flow and intuitive game design. If you would like to try the game yourself, click the link below!

AStar Dungeon Framework

Previous Next

Overview

AStar Dungeon Framework is a tool I designed to offer a simple way to add a dungeon generator to a Unity project by providing tools and much of the code that is needed to generate a simple 2D Euclidian environment. I undertook this project in order to learn more about Unity development while also trying to find an interesting and fun way to implement the knowledge I obtained from my Artificial Intelligence class (More specifically A* as a means to find the shortest path in a maze).


Project Description

For this project, I decided to implement a dungeon generator to create a dynamic dungeon and an A* shortest path finder algorithm. The dungeon generation was not the main purpose of this project; rather, the purpose of the dungeon generator was to create an interesting environment for A* to run in. First, I will discuss some of my design choices for the dungeon; then, I will discuss my implementation of A*.


For my dungeon, it is a grid-based environment that generates a 2D dungeon that is rendered in a 3D space. My dungeon generation logic can be divided into 3 total steps: generating a perfect maze, removing random edges of the maze (making it imperfect), and placing obstacles in random rooms. For the perfect maze generation, I used basic depth-first search techniques to generate a perfect mxn maze. This method of generation ensures that there is exactly one path to the end room of the maze and that every single room in the dungeon has some kind of connection to another room (no impossible-to-reach rooms). Next, I iterate through every room in the dungeon (which is stored in a dictionary) and decide whether to add extra connections from the room or not based on a variable called extraDoorProb. If we decide to add an extra connection, we pick a random unconnected neighboring room and add a door and then decide again based on extraDoorProb whether or not to add another room connection on top of that. This was a crucial design decision for the dungeon generation because it justifies the usage of A* in the first place. Meaning, if the dungeon only has one path to the exit, then an algorithm made to find the shortest path would be unnecessary. Finally, another algorithm will run through every room in the maze again and determine whether or not to add obstacles to each room based on the addObstProb variable. Adding obstacles to a room gives that room “weight,” which is important because rooms with a higher weight are rooms that A* has to try to avoid more variable rooms. All in all, these design choices were chosen so that A*’s usage in this environment could be more justified.


For the shortest path finder, I based the entire thing off of the A* pseudocode available online, which I translated to work with Unity. Essentially, once the dungeon generator has finished, it sends the shortest path finder the rooms of the starting room and the end room, and the shortest path finder uses those to find the shortest connection. For a heuristic function, I initially was just going to use Manhattan distance, but I ended up coming up with a better heuristic for this graph called weighted Manhattan distance. In short terms, I computed an estimated value for the average weight of any 1 room in my graph, and then I multiplied it by the Manhattan distance, which is more informed. This heuristic is neither admissible nor consistent, but it works better in practice because the obstacle rooms are uniformly distributed across the dungeon, making it very unlikely that the heuristic would overestimate the travel cost (Atleast that's what I believe to be the case).


Overview

This framework is available to check out on my Itch page via web player. There is a link to my Itch at the bottom of the home page under "Contact Me." This project reinforced my sculpting skills with Unity and also my knowledge on A* and tree searching algorithms.

LITE Work

Previous Next

Overview

LITE Work is a fast-paced, lighthearted minigame collection inspired by WarioWare, developed in Godot by myself and five classmates as part of our Game Design class. I served as the lead programmer, lead designer, music composer, and dialogue writer, while also taking on the project lead role. This project tested my leadership and organizational skills as I took responsibility for defining roles, setting milestones, and guiding the overall design direction. It was a challenging yet deeply rewarding experience. I felt incredibly passionate about the development process and gained invaluable insight into team dynamics, scope management, and collaborative creation.

Challenges

From the outset, I naturally stepped into a leadership role due to my ability to clearly communicate ideas and keep the project grounded in a realistic scope. Having seen previous teams (including mine) struggle with overambition, I made it a priority to keep our goals achievable while still leaving room for creativity. Immediately after our first meeting, I established our workflow by setting up a Notion board, GitHub repository, TLDraw board, and a rough draft of our Game Design Document. I began prototyping early—before we had a fully fleshed-out concept—so the rest of the team could build their contributions around a functional core. This helped avoid overlapping responsibilities and kept everyone aligned.

Balancing my leadership duties with programming and design work was the hardest part. I took on a lot (perhaps too much), but I did so out of a genuine love for game design. My top priority as a leader was to be approachable and open while also making tough decisions when ideas didn’t serve the MVP. I focused on listening more than talking, aiming to support the team while steering the project forward. If I had to identify one area for improvement, it would be delegating more. I often take on too much when I’m passionate about something, but in this case, that drive helped us produce a polished, fun game that I’m proud of.
